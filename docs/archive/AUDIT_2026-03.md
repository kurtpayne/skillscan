**Date:** 2026-03-28  
**Scope:** skillscan-security, skillscan-trace, skillscan-lint, trace-action, skillscan-corpus

## Executive Summary

The codebase is in good shape overall. No critical security issues were found. The main areas for improvement are: (1) CI job timeouts missing across all repos, (2) `analysis.py::scan()` is a 598-line monolith that should be decomposed, (3) `skillscan-trace` pyproject version is stale at `0.1.0` after tagging `v0.2.0`, (4) no dependabot or automated dependency update mechanism, and (5) several process gaps (no PR template, no CODEOWNERS). All findings are categorized by severity below.

---

## Security

### S1 — No critical issues found ✅

No hardcoded secrets, no `shell=True` in production code (the one match is a detection rule pattern, not execution), no unsafe `yaml.load()` (all use `safe_load`), no pickle usage, no path traversal vulnerabilities. The `typer.Argument(..., exists=True, readable=True)` pattern on all CLI path inputs is correct and safe.

### S2 — Dependency pinning in CI (Low) ⚠️

`pip install -e '.[dev]'` in CI is unpinned. This is acceptable for a developer tool where reproducibility is less critical than for a deployed service, but a `requirements-dev.lock` or `pip-compile` step would eliminate the risk of a transitive dependency breaking CI unexpectedly.

**Recommendation:** Add `pip-tools` to dev extras and generate a `requirements-dev.lock` file. Pin in CI with `pip install -r requirements-dev.lock`.

### S3 — No `dependabot.yml` in any repo (Low) ⚠️

No automated dependency update PRs. Given the project scans for supply-chain attacks, it should model good supply-chain hygiene itself.

**Recommendation:** Add `.github/dependabot.yml` to skillscan-security and skillscan-trace with weekly pip and GitHub Actions update schedules.

## Correctness

### C1 — `skillscan-trace` pyproject version stale (Medium) 🔴

`pyproject.toml` still shows `version = "0.1.0"` after tagging `v0.2.0`. This means `pip show skillscan-trace` and `skillscan-trace --version` will report `0.1.0` even after the v0.2.0 release.

**Fix:** Bump `version = "0.2.0"` in `skillscan-trace/pyproject.toml` and push.

### C2 — `skillscan-trace` README version stale (Low) ⚠️

README still references `v0.1.0` in the version badge and install instructions.

**Fix:** Update alongside C1.

### C3 — `scan()` function is 598 lines (Medium) 🔴

`src/skillscan/analysis.py:806 scan()` is 598 lines. This is a correctness risk as much as a quality risk — it's hard to reason about edge cases, and changes to one detection path can silently affect others. The function handles file discovery, per-file scanning, result aggregation, suppression, and output formatting.

**Recommendation:** Extract into `_discover_targets()`, `_scan_targets()`, `_apply_suppressions()`, and `_format_results()`. The `_scan_one_file()` function (254 lines) should also be decomposed.

### C4 — `rules.py::load_builtin_rulepack()` version gate uses string comparison (Low) ⚠️

The version gate compares version strings lexicographically (`user_ver < bundled_ver`). For the current `YYYY.MM.DD.N` format this is correct, but if the format ever changes (e.g. to semver), the comparison will silently break. 

**Recommendation:** Add a `_parse_rulepack_version()` helper that validates the format and raises `ValueError` on unexpected input, rather than silently falling back to string comparison.

## Code Quality

### Q1 — `analysis.py` is a 1404-line monolith (Medium) 🔴

The entire scan pipeline lives in one file. `scan()` at 598 lines and `_scan_one_file()` at 254 lines are the primary offenders. This makes the file hard to navigate and test in isolation.

**Recommendation:** Split into `analysis/scanner.py`, `analysis/targets.py`, `analysis/results.py`. Keep `analysis.py` as a thin re-export shim for backwards compatibility.

### Q2 — `cli.py::scan_cmd()` is 477 lines (Medium) 🔴

The CLI command handler contains business logic that should live in `analysis.py`. The command handler should be thin (parse args, call analysis, format output).

**Recommendation:** Move business logic from `scan_cmd()` into `analysis.scan()` or a new `analysis.scan_from_cli()` helper.

### Q3 — `type: ignore` comments are widespread but not documented (Low) ⚠️

18 `type: ignore` comments in `src/`. Most are legitimate (untyped third-party libraries: `yaml`, `torch`, `transformers`, `optimum`). However, they should carry a reason comment so future maintainers understand why the suppression is needed.

**Recommendation:** Add `# type: ignore[import-untyped]  # pyyaml has no stubs` style comments to all `type: ignore` lines.

### Q4 — `print()` in `model_sync.py` and `_corpus_manager_bundled.py` (Low) ⚠️

5 bare `print()` calls in production code. These should use the logging framework for consistency and to allow callers to suppress output.

**Recommendation:** Replace with `logger.info()` / `logger.error()` calls.

### Q5 — No `timeout-minutes` on any CI job in any repo (Medium) 🔴

If a test hangs (e.g. waiting for a network connection that never comes), CI will run for the GitHub default of 6 hours before timing out. This wastes CI minutes and delays failure notification.

**Recommendation:** Add `timeout-minutes: 15` to all test jobs across skillscan-security, skillscan-trace, and skillscan-lint.

**Recommendation:** Replace the static version badge with a dynamic PyPI badge: `[![PyPI version](https://badge.fury.io/py/skillscan-security.svg)](https://pypi.org/project/skillscan-security/)`.

### D2 — `custom-rules-format.md` version gate note is accurate ✅

The P6 version-gate note added today is correct and matches the implementation.

### D3 — Website /trace page GitHub Action section says "Coming in v0.3" (Low) ⚠️

The GitHub Action (`trace-action`) shipped today as v1. The website section should be updated from "Coming in v0.3" to "Available now".

**Recommendation:** Update the badge in `Trace.tsx` from "Coming in v0.3" to "Available — kurtpayne/trace-action@v1".

### D4 — No `CHANGELOG.md` in skillscan-security (Low) ⚠️

**Recommendation:** Add `CHANGELOG.md` with entries for v0.8.0 and prior notable releases.

## CI / Automation

### A1 — No `timeout-minutes` on CI jobs (see Q5 above) (Medium) 🔴

### A2 — Pattern-update agent runs correctly ✅

The pattern-update agent ran today (rulepack 2026.03.28.1) and correctly: (1) ran `pytest -q` before opening a PR (the new gate from the skill update), (2) updated the website rule tables, (3) did not touch the corpus or trace repos. Process is working.

### A3 — `finetune_modal.py` README bug fixed ✅

The `README.md` false-positive issue-opener bug was fixed today. The guard is in place.

### A4 — No PR template in skillscan-security (Low) ⚠️

No `.github/PULL_REQUEST_TEMPLATE.md`. PRs have no checklist to remind contributors to run tests, update docs, and check for regressions.

**Recommendation:** Add a minimal PR template with: `[ ] Tests pass`, `[ ] Docs updated if behavior changed`, `[ ] No debug code`, `[ ] CHANGELOG updated`.

### A5 — No CODEOWNERS file (Low) ⚠️

No `.github/CODEOWNERS`. For a single-maintainer project this is low priority, but it becomes important if contributors are added.

### A6 — `skillscan-trace` CI only tests Python 3.11 and 3.12 (Low) ⚠️

`skillscan-security` tests 3.12 and 3.13. `skillscan-trace` tests 3.11 and 3.12. The matrix should be consistent. Python 3.11 is approaching EOL (Oct 2027) but 3.13 support should be added to trace.

**Recommendation:** Add `"3.13"` to the skillscan-trace CI matrix.

## Process

### P1 — No formal release process documented (Low) ⚠️

The release process (bump version, tag, push, create GitHub release) is done manually and is not documented anywhere. This has led to the `pyproject.toml` version being stale (C1).

**Recommendation:** Add a `RELEASING.md` or a `scripts/release.sh` that: bumps version in pyproject.toml, runs full test suite, tags, pushes, creates GitHub release.

### P2 — Issue templates exist but are not linked from CLI output (Low) ⚠️

GitHub issue templates exist (from M10.12). The CLI `scan` output does not include a link to the feedback URL when findings are emitted. This was a planned feature.

**Recommendation:** Add a footer to the `scan` output: `Found a false positive? https://github.com/kurtpayne/skillscan-security/issues/new?template=false_positive.yml`
