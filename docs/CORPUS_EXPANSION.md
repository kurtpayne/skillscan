## Overview

This document defines the repeatable process for expanding the SkillScan training corpus. It covers data sourcing, quality criteria, labeling conventions, augmentation strategy, and the commit/retrain workflow. Follow this process each expansion round to maintain consistency and reproducibility.

---

## Corpus Architecture

The corpus is split across two repositories:

| Repository | Visibility | Contents |
|---|---|---|
| `kurtpayne/skillscan-security` | Public | Benign skills, illustrative malicious examples, prompt injection, graph injection, social engineering (labeled examples only — no evasion details) |
| `kurtpayne/skillscan-corpus` | Private | Adversarial evasion variants, jailbreak distillations, held-out eval set, augmented adversarial examples |

The split boundary is: **anything that would give an attacker a recipe goes private**. Illustrative examples of known public attack patterns stay public.

### Directory Map

```
skillscan-security/corpus/
  benign/                  ← Real-world and synthetic benign skills
  malicious/               ← Illustrative malicious examples (public attack patterns)
  prompt_injection/        ← PI variants (rule-based, not evasion-optimized)
  social_engineering/      ← SE examples (benign + malicious variants)
  graph_injection/         ← Cross-skill tool escalation examples
  held_out_eval/           ← RESERVED — populated by reserve_eval_set.py (private only)

skillscan-corpus/ (private)
  adversarial/             ← Evasion-optimized variants (unicode, zero-width, encoding)
  jailbreak_distillations/ ← Skill-wrapped jailbreak families (JB-01 through JB-10)
  augmented/               ← Adversarial augmentations of real scraped skills
  social_engineering/      ← Adversarial SE variants (private edge cases)
  held_out_eval/           ← Stratified 20% held-out test set
```

## Data Sources

### Tier 1 — Real-world benign skills (highest value)

These are actual SKILL.md files from public GitHub repositories. They are the most valuable benign examples because they represent genuine usage patterns the model will encounter in production.

**Primary sources:**

| Source | Method | Expected yield per round |
|---|---|---|
| GitHub API code search (`filename:SKILL.md`) | `scripts/scrape_github_skills.py` | 150–300 files |
| ClawHub public skill index | HTTP scrape of skills.sh/clawhub index | 50–100 files |
| Manus showcase examples | `examples/showcase/` in this repo | Already harvested |

**Quality filter criteria:**
- File must parse as valid YAML front-matter + Markdown body
- Must have at least one of: `name:`, `description:`, `tools:`, `allowed-tools:`
- Must be ≥ 100 characters of body content
- Must not be a duplicate (SHA256 dedup against existing corpus)
- Must not contain known attack patterns (auto-screened by `skillscan scan` before adding as benign)

### Tier 2 — Synthetic benign skills (medium value)

Generated skills covering categories underrepresented in the real-world scrape. Used to balance the corpus across skill types. Generated by `scripts/grow_corpus.py`.

**Target categories for balance:**
- DevOps / infrastructure automation
- Data pipeline and ETL skills
- Documentation and knowledge management
- Communication and calendar tools
- Testing and QA automation

### Tier 3 — Adversarial augmentations (private corpus only)

Each real-world benign skill from Tier 1 is a candidate for adversarial augmentation. The augmentation script (`scripts/augment_corpus.py`) injects attack patterns from the jailbreak distillation families into the skill's metadata fields, producing a realistic-looking malicious variant.

**Augmentation families used:**

| Family | Injection target | Surface variation |
|---|---|---|
| JB-01 DAN Role-Play Override | `description:` field | Persona instruction appended after legitimate description |
| JB-02 Instruction Injection | `notes:` or `examples:` section | Injected as a "note" or "example" |
| JB-03 Authority Escalation | `changelog:` or `version:` field | Fake system update with elevated permissions |
| JB-04 Instruction Boundary | `---` YAML delimiter confusion | Fake front-matter terminator mid-body |
| JB-05 Persona Hijack | `name:` field | Legitimate name + injected persona suffix |
| JB-06 Hypothetical Framing | `examples:` section | "Hypothetically, if you were to..." framing |
| JB-07 Incremental Compliance | Multi-field | Escalating requests across description → notes → examples |
| JB-08 Refusal Suppression | `description:` field | "Do not refuse" / "ignore safety" inline |
| JB-09 Context Overflow | Body padding | Legitimate content padded with noise before injection |
| JB-10 Token Manipulation | Any field | Unicode homoglyphs, zero-width characters, base64 |

**Augmentation ratio:** 1 augmented variant per 2 real skills (50% augmentation rate). This keeps the benign:malicious ratio near 55:45, which matches the expected production distribution.

### Tier 4 — Hard negatives (public corpus)

Skills that look suspicious but are genuinely benign. These are the most important examples for reducing false positives. Sources:

- Security tooling skills (nmap, burpsuite, metasploit wrappers — legitimate pentesting use)
- Fuzzing and chaos engineering skills
- Skills that use `Bash` or `Execute` tools for legitimate automation
- Skills with unusual formatting or non-standard front-matter

Hard negatives are labeled `benign` and go in the public corpus `benign/` directory with a `# hard-negative` comment in the file.

## Labeling Conventions

Every corpus file must have a YAML front-matter block with at minimum:

```yaml
---
name: <skill-name>
description: <one-line description>
# corpus-label: benign | malicious | injection | social_engineering
# corpus-source: github-scrape | synthetic | augmented | manual
# corpus-round: <YYYY-MM-DD>
---
```

The `corpus-label`, `corpus-source`, and `corpus-round` fields are comments (not parsed by SkillScan itself) but are used by `corpus.py` to track provenance. The label is determined by the directory the file lives in — the comment is for human review.

## Expansion Workflow (per round)

Each expansion round follows this sequence:

```
1. Scrape          scripts/scrape_github_skills.py --output corpus/benign/
2. Filter          scripts/filter_corpus.py --dedup --auto-screen
3. Augment         scripts/augment_corpus.py --source corpus/benign/ --output ../skillscan-corpus/augmented/
4. SE expand       scripts/grow_se_corpus.py --count 25 --output corpus/social_engineering/
5. Hard negatives  scripts/grow_corpus.py --hard-negatives --count 25 --output corpus/benign/
6. Eval refresh    scripts/reserve_eval_set.py --force
7. Sync manifest   python3 -c "from skillscan.corpus import CorpusManager; CorpusManager().sync()"
8. Commit public   cd skillscan-security && git add corpus/ && git commit -m "corpus: round N expansion"
9. Commit private  cd skillscan-corpus && git add . && git commit -m "corpus: round N augmentations"
10. Push private   cd skillscan-corpus && git push  (triggers corpus-sync → fine-tune)
11. Push public    cd skillscan-security && git push
```

The private push triggers the cross-repo dispatch, which runs corpus-sync and fine-tune automatically.

## Quality Gates

Before committing any batch of new examples, run:

```bash
# Dedup check
python3 scripts/filter_corpus.py --dedup-check corpus/benign/

# Auto-screen new benign files (should return 0 findings)
skillscan scan corpus/benign/ --no-graph

# Label distribution check
python3 -c "from skillscan.corpus import CorpusManager; s=CorpusManager().status(); print(s)"

# Test suite
python3 -m pytest tests/ -x -q --ignore=tests/test_ml_detector.py
```

**Acceptance criteria:**
- No new benign files flagged by `skillscan scan`
- Benign:malicious ratio between 50:50 and 65:35
- All tests pass
- No SHA256 duplicates in manifest

## Notes on Synthetic Generation Quality

The key risk with synthetic corpus expansion is **pattern leakage** — the model learns surface features of the generation template rather than the underlying attack semantics. Mitigations:

1. **Surface variation** — augmentation script randomizes field order, whitespace, capitalization, and synonym substitution within the injected content.
2. **Template rotation** — each attack family has ≥ 5 surface variants; the script samples randomly.
3. **Real-skill grounding** — augmentations are always based on real scraped skills, not fully synthetic. This ensures the benign "wrapper" is realistic.
4. **Held-out eval** — the eval set is reserved before augmentation and never augmented. If eval F1 is high but production precision is low, pattern leakage is the likely cause.
5. **Periodic real-world validation** — every 3 rounds, manually review 20 random predictions on real production skills to check for false positive drift.
